# HexGuard.ProblemDetails

RFC 9457 Problem Details for HTTP APIs — types, builders, and ASP.NET Core integration for producing standard, machine-readable error responses.

## Overview

`HexGuard.ProblemDetails` gives every HTTP error in your API the same RFC 9457 shape: a `type` URI,
a `title`, an HTTP `status`, a human-readable `detail`, and typed extension members. Whether the
error comes from validation, a missing resource, or an unhandled exception, clients see a consistent
`application/problem+json` response.

The package targets `net10.0` and depends only on `Microsoft.AspNetCore.App` (the shared framework).

## Installation

```shell
dotnet add package HexGuard.ProblemDetails
```

## Quick Start

### 1. Register middleware (optional but recommended)

```csharp
using HexGuard.ProblemDetails;

var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

app.UseMiddleware<ProblemDetailsMiddleware>();

app.MapGet("/api/products/{id}", (int id) =>
{
    // Thrown ProblemDetailsException is caught by middleware
    var product = db.Products.Find(id)
        ?? throw new ProblemDetailsException(new ProblemDetailsBuilder()
            .WithType(WellKnownProblemTypes.NotFound)
            .WithTitle("Resource Not Found")
            .WithStatus(404)
            .Build());

    return Results.Ok(product);
});

app.Run();
```

### 2. Return Problem Details from Minimal APIs

```csharp
app.MapGet("/api/validation-error", () =>
{
    var pd = new ProblemDetailsBuilder()
        .WithType(WellKnownProblemTypes.ValidationError)
        .WithTitle("Validation Error")
        .WithStatus(400)
        .WithDetail("The request contains invalid fields.")
        .WithExtension("errors", new[]
        {
            new { field = "name", code = "required", message = "Name is required." },
        })
        .Build();

    return pd.ToProblemResult();
});
```

## API Reference

### `ProblemDetails` record

A sealed record with optional properties matching RFC 9457:

| Member       | Type                               | Description                                             |
| ------------ | ---------------------------------- | ------------------------------------------------------- |
| `TypeUri`    | `string?`                          | URI reference identifying the problem type              |
| `Title`      | `string?`                          | Short human-readable summary (stable per type)          |
| `Status`     | `int?`                             | HTTP status code generated by the origin server         |
| `Detail`     | `string?`                          | Human-readable explanation for this occurrence          |
| `Instance`   | `string?`                          | URI identifying this specific occurrence                |
| `Extensions` | `IReadOnlyDictionary<string, object?>?` | Additional members (validation errors, trace IDs)  |

Default-constructed instances have all null properties. The status defaults to `500` when used
with middleware or `ToProblemResult()`.

### `ProblemDetailsBuilder`

Fluent builder that guards string setters against `null` and produces immutable extension
dictionaries on `Build()`.

```csharp
var pd = new ProblemDetailsBuilder()
    .WithType(WellKnownProblemTypes.ValidationError)
    .WithTitle("Validation Error")
    .WithStatus(400)
    .WithDetail("The request contains invalid fields.")
    .WithInstance("/api/products")
    .WithExtension("traceId", Activity.Current?.Id)
    .Build();
```

| Method                           | Null guard  | Notes                                               |
| -------------------------------- | ----------- | --------------------------------------------------- |
| `WithType(string)`               | throws      |                                                     |
| `WithTitle(string)`              | throws      |                                                     |
| `WithStatus(int)`                | none        |                                                     |
| `WithDetail(string)`             | throws      |                                                     |
| `WithInstance(string)`           | throws      |                                                     |
| `WithExtension(string, object?)` | key throws  | Overwrites previous value for the same key          |

### `WellKnownProblemTypes`

Pre-defined type URIs for common error categories:

| Constant              | URI                                                                    |
| --------------------- | ---------------------------------------------------------------------- |
| `AboutBlank`          | `about:blank`                                                          |
| `ValidationError`     | `https://docs.hexguard.dev/problems/validation-error`                  |
| `NotFound`            | `https://docs.hexguard.dev/problems/not-found`                         |
| `OutOfRange`          | `https://docs.hexguard.dev/problems/out-of-range`                      |
| `BadRequest`          | `https://docs.hexguard.dev/problems/bad-request`                       |
| `InternalServerError` | `https://docs.hexguard.dev/problems/internal-server-error`             |

### `ProblemDetailsException`

An exception that carries a `ProblemDetails` payload. The middleware catches it automatically
and writes the embedded `Details` as the response.

```csharp
throw new ProblemDetailsException(new ProblemDetailsBuilder()
    .WithType(WellKnownProblemTypes.NotFound)
    .WithTitle("Resource Not Found")
    .WithStatus(404)
    .Build());
```

The exception message defaults to `Details.Title`, falling back to `Details.Detail`.

### `ProblemDetailsMiddleware`

ASP.NET Core middleware that catches exceptions and writes `application/problem+json` responses.

```csharp
app.UseMiddleware<ProblemDetailsMiddleware>(new ProblemDetailsMiddlewareOptions
{
    CatchAllExceptions = true,       // default: true
    IncludeExceptionDetails = false,  // default: false; enable in Development
});
```

**Behavior matrix:**

| `CatchAllExceptions` | `IncludeExceptionDetails` | Exception                 | Response                                        |
| -------------------- | ------------------------- | ------------------------- | ----------------------------------------------- |
| `true` (default)     | `false` (default)         | `ProblemDetailsException` | Uses the exception's `Details` payload          |
| `true`               | `false`                   | Any other `Exception`     | Generic 500, `"Internal Server Error"`          |
| `true`               | `true`                    | Any other `Exception`     | 500 with exception type name and message        |
| `false`              | any                       | `ProblemDetailsException` | Uses the exception's `Details` payload          |
| `false`              | any                       | Any other `Exception`     | Falls through to ASP.NET Core's default handler |

### `ProblemDetailsResultExtensions`

Extension method for returning Problem Details from Minimal API endpoints:

```csharp
return new ProblemDetailsBuilder()
    .WithType(WellKnownProblemTypes.BadRequest)
    .WithTitle("Bad Request")
    .WithStatus(400)
    .Build()
    .ToProblemResult();
```

## Serialization

All Problem Details responses use `System.Text.Json` with:

- **CamelCase property naming** — e.g., `TypeUri` → `typeUri`
- **Null omission** — properties with `null` values are excluded from the JSON body

Extension members are serialized as top-level JSON properties (no `"extensions"` envelope),
following the RFC 9457 specification.

```json
{
  "typeUri": "https://docs.hexguard.dev/problems/validation-error",
  "title": "Validation Error",
  "status": 400,
  "detail": "The request contains invalid fields.",
  "instance": "/api/products",
  "errors": [
    { "field": "name", "code": "required", "message": "Product name is required." }
  ]
}
```

### `ValidationResultExtensions` — simplified validation responses

Extension method that converts a `HexGuard.ValidationContracts.ValidationResult` into a complete
RFC 9457 response in one call. No manual `Results.Problem()` unpacking needed.

```csharp
using HexGuard.ProblemDetails;
using HexGuard.ValidationContracts;

app.MapPost("/api/products", (ProductPayload payload) =>
{
    var builder = new ValidationResultBuilder();

    if (string.IsNullOrWhiteSpace(payload.Name))
        builder.AddError("name", ValidationErrorCode.Required, "Product name is required.");

    if (payload.Price <= 0)
        builder.AddError("price", ValidationErrorCode.OutOfRange, "Price must be positive.");

    var result = builder.Build();
    if (result.IsValid)
        return Results.Ok(new { saved = true });

    // One call — returns application/problem+json with errors in the "errors" extension
    return result.ToProblemResult(
        statusCode: 400,
        detail: "The request payload failed validation.",
        instance: "/api/products");
});
```

## Error Handling Strategies

| Pattern                          | When to use                                                   |
| -------------------------------- | ------------------------------------------------------------- |
| Middleware only                  | You want unhandled exceptions to produce Problem Details      |
| `ToProblemResult()`              | You control the endpoint and want to return without throwing  |
| `ProblemDetailsException`        | Business/service-layer code needs to signal an error upward   |
| `ValidationResult.ToProblemResult()` | Validation endpoints — one call from `ValidationResult` to response |

Using all four together is common — validation endpoints return via `ToProblemResult()`, other
endpoints use `ProblemDetails.ToProblemResult()`, service code throws `ProblemDetailsException`
for exceptional paths, and the middleware provides a safety net for anything that slips through.
